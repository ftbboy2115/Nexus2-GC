# Investigation: Sim Engine P&L Discrepancy (MLEC Batch Test)

## Background

Running the MLEC batch test case (`ross_mlec_20260220`) revealed multiple anomalies that **may indicate systemic bugs** in the sim engine, not just this one test case.

## Observed Anomalies

### 1. P&L Math Doesn't Match Entry/Exit Prices
```
entry_price: 19.96
exit_price: 18.91
shares: 1,302
reported_pnl: +$244.19
```
**Expected**: Long position at $19.96 exiting at $18.91 = (18.91 - 19.96) × 1,302 = **-$1,367 loss**
**Reported**: +$244.19 profit

### 2. Timestamps Are Wrong
```
entry_time: 2026-02-20T23:16:57Z  (6:16 PM ET)
exit_time:  2026-02-20T23:16:58Z  (6:16 PM ET)
```
- These are **after market hours** (market closes 4 PM ET)
- Entry-to-exit is **1 second** — not realistic for a pattern-based trade

### 3. Trade Details May Not Reflect Actual Fills
- Entry trigger `whole_half_anticipatory` — anticipating $20 break
- Exit reason `mental_stop` — stopped out at $19.46 consolidation low
- But exit price shows $18.91, well below the stop

## Open Questions for Investigation

1. **Where does the sim engine calculate P&L?** Is it using the reported entry/exit prices, or are there intermediate fills being aggregated?
2. **Are entry_price/exit_price in the trade detail the FIRST fill / LAST fill, or volume-weighted averages?** The disconnect between displayed price and P&L suggests these might not be the actual fill prices used in P&L calc.
3. **Where do the timestamps come from?** Are they generated by `datetime.utcnow()` at the time the sim runs (wall clock), or from the bar data's simulated market time?
4. **Is exit_price correctly reflecting the stop price?** The stop is at $19.46 but exit shows $18.91 — could this be slippage logic, or a bug?
5. **Do OTHER test cases show similar P&L discrepancies?** Run the full batch suite with `include_trades: true` and compare entry/exit price math vs reported P&L.

## Investigation Approach

### Phase 1: Backend Planner — Code Research
Trace the sim engine code path for:
- P&L calculation (where does `realized_pnl` come from?)
- Trade detail serialization (what populates `entry_price`, `exit_price`, `entry_time`, `exit_time`?)
- Timestamp source (wall clock vs simulated market time?)

**Key files to investigate** (starting points — verify existence):
- `nexus2/domain/automation/warrior_engine.py` — may contain sim logic
- `nexus2/domain/simulation/` — sim engine directory if it exists
- Search for `realized_pnl`, `entry_price`, `exit_price` across the codebase

### Phase 2: Testing Specialist — Cross-Case Validation
- Run `run_batch_concurrent` with ALL test cases and `include_trades: true`
- For each trade: verify that (exit_price - entry_price) × shares ≈ reported P&L
- Flag any cases where the math doesn't match
- Check if timestamps are always wall-clock or sometimes simulated

### Phase 3: Remediation
Based on findings, fix the root cause(s).
